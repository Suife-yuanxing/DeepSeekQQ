## 审计方法

对差距分析文档中每一项声明的"Bot 当前逻辑"，逐一在源代码中验证：
- ✅ = 分析准确，代码确实如此
- ⚠️ = 分析方向正确但细节有偏差
- ❌ = 分析有误，与代码不符
- ➕ = 代码审查中发现的补充问题

**补充发现**：

➕ **audit-2-1**：`EmotionState` 的 `confidence` 字段从关键词匹配中获取（[emotion_classifier.py:22-58](DeepSeekQQ/plugins/deepseek/emotion_classifier.py#L22-L58)），但即使关键词置信度高达 0.9，代码仍会走后续 LLM 分析（因为 `context_analyzer.py` 中 LLM 分析独立运行）。这意味着：一句"滚"同时触发关键词 `angry=0.9` 和 LLM 情绪分析——双重计算浪费 API 调用。**严重度: 中**

➕ **audit-2-2**：差距分析指出"非语言触发缺失"，但检查代码发现 `conversation_fatigue.py` 已经实现了回复变短检测、回复速度变慢检测。这些信号没有反馈给情绪系统——用户的「回复变短+间隔拉长」只影响疲劳判定，不影响 bot 情绪。这是跨模块断裂的典型案例。**严重度: 高**

## 五、承诺追踪审计

| 差距分析声明 | 代码验证 | 证据 |
|-------------|---------|------|
| 正则匹配 "明天/下次/回头/等下/晚点" | ✅ **确认** | [promise_tracker.py:20-26](DeepSeekQQ/plugins/deepseek/promise_tracker.py#L20-L26): 5 个模式 |
| 固定偏移（明天=+24h, 回头=+1-6h） | ✅ **确认** | [promise_tracker.py:102-113](DeepSeekQQ/plugins/deepseek/promise_tracker.py#L102-L113): `estimate_due_time` |
| 固定 20% 遗忘率 | ✅ **确认** | [promise_tracker.py:38](DeepSeekQQ/plugins/deepseek/promise_tracker.py#L38): `_FORGET_RATE = 0.20` |
| 兑现前缀模板 | ✅ **确认** | [promise_tracker.py:51-56](DeepSeekQQ/plugins/deepseek/promise_tracker.py#L51-L56): `_PROMISE_FULFILL_TEMPLATES` |

➕ **audit-5-1**：`extract_promises` 的正则存在漏抓——模式 `r'明天(?:再|就|帮)?(.{2,20}?)(?:吧|哦|啦|哈|~|！|。|$|，)'` 要求承诺内容紧跟"明天再/就/帮"，但真人说法是"明天我去查一下告诉你"——这里"明天"和承诺内容之间插入了主语+动词，正则抓不到。例如"明天我帮你看一下"中的"我"和"帮你看"让 `. {2,20}` 捕获了意外内容。**严重度: 中**

➕ **audit-5-2**：遗忘逻辑是纯二元的（忘或不忘），但真人的遗忘是渐进的——先记得但拖延、再模糊、最后真忘。当前实现抹平了这个层次。**严重度: 低**

➕ **audit-5-3**：`_FORGIVEN_WINDOW = 86400 * 3`（3 天），但 `get_forgotten_to_apologize` 查询条件是 `due_at <= now AND due_at >= now - _FORGIVEN_WINDOW`，即只回溯过去 3 天。如果 bot 因为某些原因 3 天内没发消息，道歉窗口就关闭了——真人不会因为"过了道歉窗口"就不道歉了。**严重度: 低**

➕ **audit-6-3**：微事件池的 `register_micro_events()` 允许外部追加（[behavior_engine.py:649-656](DeepSeekQQ/plugins/deepseek/behavior_engine.py#L649-L656)），但生成的微事件没有持久化——重启后追加的事件丢失。**严重度: 极低**

## 八、架构结构性审计

### 8.1 模块间断裂面

差距分析正确地指出了各模块独立决策的问题。以下是通过代码审查发现的**具体断裂点**：

| 断裂面 | 模块 A | 模块 B | 缺失的链接 |
|--------|--------|--------|-----------|
| `can_interrupt` 字段无人读取 | `activity_sim.py` | `handler.py` / `stage_llm.py` | 活动不可中断 → 回复超时/跳过 |
| 情绪不反馈给疲劳 | `emotion_deep.py` | `conversation_fatigue.py` | bot 生气了 → 故意冷淡 → 不需要收尾 |
| 疲劳不反馈给早晚安 | `conversation_fatigue.py` | `night_reminder.py` | 对话已自然结束 → 不需要晚安 |
| 活动不联动回复速度 | `activity_sim.py` | `stage_llm.py` | 打游戏 → 回复延迟/简短 |
| 作息不联动早安 | `schedule.py` | `morning_greeting.py` | 虚拟起床时间 → 决定何时发早安 |
| 非语言信号不进情绪 | `conversation_fatigue.py` | `emotion_deep.py` | 回复变短 → bot 感到被冷落 |
| 承诺不感知疲劳 | `promise_tracker.py` | `conversation_fatigue.py` | 对话快结束了 → 不要在此时兑现承诺 |

### 8.2 全局状态碎片化

搜索代码库发现以下全局/模块级可变状态：

- `proactive/morning_greeting.py:26` — `_consecutive_morning_skips: dict`
- `schedule.py:22-23` — `_daily_offset_cache: dict`, `_offset_date: str`
- `behavior_engine.py:543` — `_hot_topic_cache: List`
- `behavior_engine.py:625` — `_MICRO_EVENTS: List`
- `behavior_engine.py:485` — `_cached_special_dates: Optional[Dict]`
- `emotion_classifier.py:90-100` — `EmotionBuffer` 实例（每个会话一个）
- `proactive/__init__.py:41-42` — `_scheduler`, `_registered_bot_id`

虽然有 `global_state.py`（2026-06-18 创建），但以上模块均未迁移到它。`global_state.py` 创建后仅被自身引用。**严重度: 中**

### 8.3 时间基准不统一

| 模块 | 获取时间的方式 | 问题 |
|------|--------------|------|
| `schedule.py` | `datetime.now()` + 虚拟偏移 | 有偏移但只影响 schedule 自身 |
| `morning_greeting.py` | `datetime.now()` | 用系统真实时间，忽略 schedule 偏移 |
| `activity_sim.py` | `time.time()` (隐式) | 按时段切换但不考虑 schedule 的偏移 |
| `behavior_engine.py` | `datetime.now()` | 季节/节日判断用系统真实时间 |

没有一个统一的"bot 感知时间"——每个模块各自读系统时钟。这意味着 schedule 虽然模拟了"今天是周末所以晚起"，但其他模块完全感知不到这个偏移。**严重度: 中**

## 九、补充发现（差距分析未覆盖的问题）

### audit-new-1：`_generate_proactive_message` 中的提示词与行为引擎输出的冲突

[shared.py:222-239](DeepSeekQQ/plugins/deepseek/proactive/shared.py#L222-L239) 的 system prompt 规则 10 要求「情绪表达藏在字里行间」，但行为引擎注入的 hint 经常是「你突然想撒娇，语气变软」（来自 `get_mood_swing`）。两个指令矛盾——一个要求含蓄，一个要求直接表达。LLM 的处理结果不可预测。**严重度: 低**

### audit-new-2：好感度数据源分散

好感度在以下多个地方以不同方式读取：
- `shared.py:108` — `get_affection(user_id)` 返回 `{"score": ..., "title": ...}`
- `morning_greeting.py:154` — 只用 `get_last_greeting_time` 不直接查好感度
- `emotion_deep.py:186` — `affection_score` 作为参数传入
- `behavior_engine.py:270` — `affection_score` 作为参数传入
- `personality_drift.py:88` — `affection_score` 作为参数传入

各处传参方式不统一（有的从 DB 查，有的从 `get_affection` 查），增加了数据不一致的风险。**严重度: 低**

### audit-new-3：`shared.py` 每次生成主动消息前都调天气 API

[shared.py:189-194](DeepSeekQQ/plugins/deepseek/proactive/shared.py#L189-L194) 在 `_generate_proactive_message` 中每次都 `await get_weather()`，即使这个场景不需要天气信息（如 checkin 场景）。虽然有缓存，但这违反了按需获取原则。**严重度: 极低**

### audit-new-4：`schedule.py` 的 `_get_virtual_now` 未真正实现虚拟时间

[schedule.py:43-50](DeepSeekQQ/plugins/deepseek/schedule.py#L43-L50): `_get_virtual_now()` 函数签名暗示要返回带偏移的虚拟时间，但实际实现中只是 `return now`（返回系统真实时间）。注释说"用虚拟时间做所有时间段判断"，但实际上偏移量在 `get_schedule_state` 中手动加减，而非统一的虚拟时间函数。**严重度: 低**
