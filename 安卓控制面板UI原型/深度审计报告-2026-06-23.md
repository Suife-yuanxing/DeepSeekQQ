### 3.2 🔴 脚本加载 Bug — 数据层完全失效 — 2 页

**QQ通道.html**（362行）和**微信通道.html**（326行）存在相同的致命 bug：

```html
<!-- QQ通道.html L327 -->
<script src="shared/app.js">
// Load QQ channel status          ← 这段代码永远不会执行！
if (API.isLoggedIn()) {
  function loadQQStatus() {
    API.getQQStatus().then(...)    // 真 API 调用存在，但无法执行
    API.getQQStats().then(...)
    API.getQQRecentMessages(3).then(...)
  }
  loadQQStatus();
  setInterval(loadQQStatus, 30000);
}
</script>
```

```html
<!-- 微信通道.html L308 -->
<script src="shared/app.js">
// Load WeChat channel status     ← 同样永远不会执行！
if (API.isLoggedIn()) {
  function loadWxStatus() {
    API.getWechatStatus().then(...)
  }
  loadWxStatus();
}
</script>
```

**HTML 规范**：当 `<script>` 标签同时有 `src` 属性和内联内容时，浏览器**只加载外部脚本，忽略内联代码**。这是自 1999 年 HTML 4.01 以来的标准行为。

**影响**：
- QQ通道.html 的 `loadQQStatus()` 永不被调用 → 页面显示静态占位 HTML（"已连接" / "0" / 硬编码消息），虽然 `API.disconnectQQ()` 在断开按钮的 onclick 中（可独立触发），但状态数据永不刷新
- 微信通道.html 的 `loadWxStatus()` 永不被调用 → 绑定状态永显静态 "未绑定"，但 `API.bindWechat()` 绑定按钮可独立触发

**修复**：将内联代码移到独立 `<script>` 标签（无 `src` 属性），或移到 `shared/app.js` 末尾。

### 3.4 🟡 编辑个人资料 — 函数覆盖时序脆弱

编辑个人资料.html 的原始 `saveProfile()`（L186）只做本地 toast 提示，不调 API。L219 行用 `window.saveProfile = function() { ... API.updateProfile() ... }` 覆盖。但保存按钮是 `onclick="saveProfile()"`——如果覆盖脚本在按钮被点击之前执行（通常如此），则正常工作；但如果用户在页面完全加载前点击，则调用的是无 API 的原始版本。**时序脆弱，建议直接将 API 调用写入原始 `saveProfile()`。**

## 八、总体评价

**后端实现质量：A-（优秀）**。76 端点全部真实存在，代码结构清晰（13 个独立 router 模块），测试覆盖扎实（1318 全绿），鉴权体系完整（JWT + ownership + admin guard），部署链路可运转（systemd + git push 部署）。

**前端对接质量：B-（良好，但验收不严谨）**。"19/19 全部接通"的宣称过于乐观——实际 12/16 功能页完全接通（75%），2 页存在 HTML 规范级脚本加载 bug（QQ/微信通道页数据层完全失效），1 页 5/8 模块仍硬编码（管理员面板）。这些问题均可修复，单页成本 <5 分钟。

**总体评价：方向正确、骨架扎实、核心链路可运转，但前端验收存在系统性 overclaim——建议建立"逐页 API 调用 checklist"机制，避免文档层面的"全部接通"掩盖代码层面的具体问题。**
