## 二、🔴 严重发现（7 项 — 必须修正才能开工）

### S1. 1.6 WS 聊天的 "api.py 已支持流式" 声明不成立

**实际代码**（`api.py:79`）：
```python
payload = {"model": ..., "stream": False, ...}
async def call_deepseek_api(...) -> str:  # 返回完整字符串，非 async generator
```

清单声称 `call_deepseek_api` 已支持流式，实际 `stream: False`，返回 `str`。要支持 WS 逐 token 流式，需新写 SSE 流式解析函数（`aiohttp` 流读 + `data: {...}` 帧解析 + 逐 chunk yield）。

**真实成本**：+1.5d，应计入 1.6。3.5d → **5d**。

### S2. 1.6 WS 聊天的 "Pipeline 可复用" 是架构级决策，清单未暴露

**实际代码**（`pipeline.py:170`）：
```python
async def handle_chat(bot: Bot, event: MessageEvent)
```
`ChatContext` 的 `bot: Bot` 和 `event: MessageEvent` 是必填，22 个 stage 大量访问 `ctx.event.user_id`、`ctx.bot.send()`、`ctx.event.get_message()` 等 NoneBot2 API。**Pipeline 绑死 NoneBot2 事件循环**，FastAPI WS handler 没有这些对象。

两个选项：
- **方案 A（适配层）**：构造假 `Bot`/`MessageEvent` 适配器，+3-5d，22 个 stage 都要兼容测试
- **方案 B（绕过 Pipeline）**：只调 `call_deepseek_api` + `db_memories`，+0d，但 **App 聊天丢失情绪/记忆/人设演化/好感度/缺席事件等全部增强能力**——App 体验远不如 QQ 聊天

**这是架构级决策，清单当作"复用"轻描淡写。** 建议实施前先做技术 spike 验证方案 A 可行性。

### S3. 1.6 db_memories 无法支撑 client_id 幂等去重

**实际代码**（`database.py:235`）：`memories` 表结构 `(id, session_id, role, content, timestamp, archived)`——**无 `client_id`、无 `bot_id`、无 `channel`、无 `status`、无 `retry_count` 字段**。

清单声称 db_memories 支撑幂等去重，实际依赖 Phase 0.11 的 `chat_messages` 新表（2d，已在 Phase 0）。1.6 实际依赖 Phase 0.11 完成才能开工，清单未标注此前置。

### S4. 1.17 通道状态 "NapCat 连接状态可复用" 跨进程不可达

**实际代码**：NapCat 连接状态通过 `nonebot.get_bots()`（`startup.py:132/156/210`）获取，`/admin/api/heat` 读 `heat_manager._groups`——两者都是 **NoneBot2 进程内内存**。FastAPI 后端是**独立 systemd 进程**（8766），与 NoneBot2 进程（8082）不共享内存。

跨进程获取状态需要：(a) NoneBot2 暴露 HTTP 端点供 FastAPI 查询，或 (b) 共享 Redis/SQLite 心跳表，或 (c) 合并进程。清单完全没提及。**真实成本**：+0.5-1d，1.17 的 1d → **1.5-2d**。

### S5. WS JWT 通过 URL query param 传递 — Token 泄露高风险

开发计划 Section 5.4.1 写 `ws[s]://host:8766/ws?token=<jwt_access_token>`，清单 1.6 只说"JWT 认证"未否定此方案。URL query param 会被写入 nginx access log、systemd journal、uvicorn access log、浏览器 history、Referer header。access_token(15min) 泄露后攻击者可在 15 分钟内完全冒充用户。

**修复**：WS 握手用 `Sec-WebSocket-Protocol` 子协议头或 `Authorization: Bearer` header 传递 JWT；若必须用 query param，需关闭 uvicorn access log 对 WS 路径的记录 + 自定义日志脱敏。

### S6. 单一 ADMIN_API_KEY 无法区分 admin Token 与 user Token — 鉴权架构断裂

`middleware/auth.py` 是全局共享的 `ADMIN_API_KEY` 字符串（明文 `==` 比较），`is_admin_path()` 只匹配 `/admin/*`，不匹配 `/api/v1/admin/*`。多租户后普通用户持有 JWT（含 user_id），不是 ADMIN_API_KEY。现有中间件无 JWT 验证、无 user_id 提取、无 is_admin 判断。

**修复**：新建 JWT 验证中间件（python-jose），`/api/v1/admin/*` 要求 `is_admin=1`，`/api/v1/*` 用户路由提取 `user_id` 做所有权校验。现有 ADMIN_API_KEY 保留保护 8082 端口的 `/admin/*`，两套认证物理隔离。

### S7. 数据权限 6 开关存储了但无执行点 — 隐私合规空转

清单 1.2 存 `ai_training / learn_chat_style / remember_interests / usage_statistics / crash_report / third_party_sharing` 6 开关，但两份文档均未描述执行点：`ai_training=false` 后消息在哪行代码被排除出训练集？`remember_interests=false` 后 `db_preferences` 是否停止写入？`third_party_sharing=false` 后哪些响应被过滤？

**App 上架硬要求**——只存开关不执行等于合规空转，法务审查不过。

**修复**：每个开关明确执行点（Pipeline 各 stage 加 `if user_data_permissions['xxx']` 分支），在 Phase 0.11 建表时同步落地。

P0 声明只需 1.2+1.4+1.6 精简版 5d 跑通"注册→创建Bot→WS聊天"，但缺：

### H4. Phase 0 最小子集可跳过部分任务

Phase 0 全量 34-39d，但对 Phase 1 的硬前置子集仅 **8.5d**：
- 0.11 建表（2d）+ 0.1-0.4 硬阻塞修复（2.5d）+ 0.5 API Key 路由（2d）+ 0.6 LLM 队列（1d）+ 0.9 session_id 第 1 批（1d）

可跳过先做 Phase 1 验证：0.7a（CausalContext LRU）、0.10（session_id 第 3 批）、0.13a（数据迁移 SQL）、0.13b（备份演练）、0.14（全量回归）。**但 0.11 + 0.1-0.6 核心链路不可跳。**

### H5. 所有权校验全部端点未声明

清单的 `bot_id` / `message_id` / `stats/{bot_id}` 端点均未在验收点写明"校验资源属于当前 JWT user_id"。bot_id 是连续整数，A 用户可猜 B 用户的 bot_id 越权访问。涉及端点：
- 1.4：`GET/PUT/DELETE /bots/{id}`、`POST /bots/{id}/avatar`、`DELETE /bots/{id}/memory`
- 1.6：`GET /messages?bot_id=`、`DELETE /messages/{id}`、`/recall`、`/report`
- 1.7：`PUT /bots/{id}/abilities`（可窃取他人 LLM 额度）
- 1.16：8 个 `/stats/{bot_id}/*`（可探测他人好感度等隐私）
- 1.17：`/channel/qq/recent-messages`（可越权看他人 QQ 消息）

**修复**：每个端点验收点显式加"校验 `bot_configs.user_id == JWT.user_id`，不符返回 403 `bot_not_owned`"。

### H6. OAuth 微信 callback 缺 state 参数 — CSRF 窗口

清单 1.2：QQ callback 是 `{code, state}`，微信 callback 是 `{code}`——微信缺 state。无 state 无法防 OAuth CSRF，攻击者可将自己的微信 code 注入受害者会话。

**修复**：微信 callback 也必须校验 state（session 绑定的随机 nonce）。

### H7. /auth/logout 无服务端吊销机制

清单写"吊销 refresh"，但无 `revoked_refresh_tokens` 表设计。若 refresh 只标记"已使用"而非"已吊销"，被窃取的 token 在 30s 宽限期内仍可刷新。

**修复**：新建 token 黑名单表，logout 时立即吊销当前 refresh 的 jti（无宽限期）。

### H8. refresh token TTL 不一致

清单 1.2 写 `refresh(30d)`，开发计划 Section 5.4.1 写 `refresh(7d)`。30 天被窃取后攻击窗口是 7 天的 4 倍。**需统一为 7d**。

### H9. 限流未与具体端点绑定

Phase 0.12 定义了限流策略（Token 30/分 + 用户 10/分 + IP 60/分），但清单 ~80 端点无一处提及限流绑定。`/auth/sms` 无限流可被短信轰炸，`/auth/login` 无限流可暴力枚举验证码，`/chat/ws` 无限流可 DoS。

## 四、🟡 中优先级发现（8 项）

### M1. 编辑个人资料头像 — 原型与清单语义冲突

原型 `changeAvatar()` 实际是**轮换猫娘 CSS 变体**（gentle→tsundere→energetic→默认），**无 `<input type="file">`，无真上传**。清单假设 `POST /user/avatar` multipart 上传。需决策：原型补 file input，或清单补"预设头像选择"端点。

### M2. 聊天背景/主题/字号 — 原型 localStorage vs 清单后端持久化

原型用 `localStorage` 存聊天背景 base64、主题、字号，换设备不同步。清单有 `GET/PATCH /user/settings` 端点但原型未对接。**需统一**：原型改为调后端，或接受多设备不同步。

### M3. API Key 管理原型 — 完整 key 内嵌 HTML

原型 `onclick="copyKey('sk-prod-8a3f2...c91d', this)"` 前端可见完整 key，与清单"前端永远不返回完整 key"直接矛盾。复制功能应仅复制 suffix 或一次性展示后不可再获取。

### M4. 话题排行无现成资产

`topic_tracker.py` 是**纯内存**实现（重启丢失），`memories` 表无 topic 字段。三个选项：(a) 批量 LLM 分类历史消息（成本高）、(b) 新建 `message_topics` 表 + 实时分类（需改 Pipeline）、(c) 内存分类日聚合（重启丢失）。**确认是真风险，需 +1-2d。**

### M5. 成就墙零现成资产

全项目 grep `achievement` **零匹配**。需新建定义表 + 解锁逻辑 + 9 个具体成就定义（初次见面/连续 7 天/1000 条消息/好感 Lv.5/10/聊 5 话题/连续 30 天/一起过生日/年度用户）。**+1-2d。**

### M6. 数据库双写/迁移策略缺失

现有 `memories` 表（`database.py:235`）vs 计划 `chat_messages` 表（Section 7.2）。两份文档未说明：App 通道消息写哪张表？QQ 通道消息是否仍写 memories？历史数据是否迁移？`db_memories` 改造还是新建并行模块？Pipeline 的 `save_message` 调用点是否分流？

### M7. 两套 FastAPI 进程 DB 共享未说明

现有 `web_admin.py` 在 8082（NoneBot2 `driver.server_app`），新建 `api_server.py` 在 8766（独立 FastAPI）。两者共享同一批 SQLite DB 文件。WAL 模式下多进程读写安全，但清单未说明 8766 如何获取 DB 路径、是否复用 `db_core.py` 连接池、两进程同时写的 `busy_timeout` 是否足够。

### M8. 敏感词过滤覆盖面不足

清单 1.15 只覆盖 Bot 名 + 聊天内容，漏了：用户昵称、签名 bio、黑名单原因、Bot 口头禅 catchphrase、Bot 背景故事 backstory。这些都是用户可见自由文本。
