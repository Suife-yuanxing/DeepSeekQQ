- 复用：无（纯新建 users 表 + JWT 双 Token）
- **v2 关键设计**：
  - 手机号 bcrypt + **AES-256-GCM**（每条独立 IV + KMS Envelope Encryption，对齐 Section 7.2）
  - JWT access(15min) + **refresh(7d)**（v2: 从 30d 改 7d，H8），Dio Interceptor 无感刷新
  - **refresh token 轮换 + 黑名单**：刷新时旧 refresh 失效；新建 `revoked_tokens` 表存 jti；logout 立即吊销无宽限期；刷新先查黑名单（H7）
  - **OAuth state CSRF 防护**：QQ + 微信 callback 都校验 state（H6）
  - data_permissions 6 开关与原型 [数据权限.html](数据权限.html) 一一对应（以前端 6 项为准，扩展计划 Section 7.2 的 3 项）
  - **S7 数据权限执行点**（必须在 Phase 0.11 建表时同步落地，否则合规空转）：
    - `ai_training=false` → Pipeline 不将该用户消息加入 embedding 训练集
    - `learn_chat_style=false` → `db_preferences` 停止写入该用户
    - `remember_interests=false` → `memory_tags` 停止写入该用户
    - `usage_statistics=false` → `token_tracker` 不上报该用户调用
    - `crash_report=false` → `error_reporter` 不附带该用户上下文
    - `third_party_sharing=false` → admin 聚合 API 排除该用户
- 前端触发：**2.2 登录/注册** 可开工

### Task 1.6 — WebSocket 聊天 + 消息 API（**5d**）→ 对接 3 个原型 ⭐核心

> **v2 修正**：3.5d → 5d（S1 流式重构 +1.5d；S2 架构决策见下；S3 硬依赖 Phase 0.11；S5 WS token 传递方式）
>
> **S2 架构级决策（必须先做 spike）**：
> - 现状：`pipeline.py:170 handle_chat(bot: Bot, event: MessageEvent)` 绑死 NoneBot2，22 个 stage 访问 `ctx.bot.send()` / `ctx.event.user_id` 等 NoneBot2 API。FastAPI WS handler 无这些对象。
> - **方案 A（适配层，+3-5d）**：构造假 `Bot`/`MessageEvent` 适配器，App 聊天拥有情绪/记忆/人设演化/好感度/缺席事件全部增强能力。**完整版 Phase 1.6 用此方案**，实施前先做 spike 验证。
> - **方案 B（绕过 Pipeline，+0d）**：只调 `call_deepseek_api` + `db_memories`，App 聊天功能大幅缩水（无情绪/记忆/人设）。**P0 三件套先用此方案快速跑通**，后续升级到方案 A。

| 端点 | 方法 | 字段 | 对接原型 |
|------|:---:|------|---------|
| `/api/v1/chat/ws` | WS | **S5: JWT 通过 `Sec-WebSocket-Protocol` 子协议头传递，禁用 URL query param**；帧：`{type:"msg", bot_id, text, client_id}` → 流式 `{type:"token", text}` / `{type:"done", server_id}` / `{type:"typing"}` / `{type:"status", msg_id, state}` / **v2 补 `{type:"read", msg_ids:[]}` 批量已读帧** | 聊天页（流式） |
| `/api/v1/messages` | GET | `?bot_id=&cursor=&limit=50` → `[{id, role, content, type, time, status}]` **H5: 校验 bot_id 属于 JWT.user_id** | 聊天页历史、聊天记录 |
| `/api/v1/messages/search` | GET | `?q=&bot_id=&date_from=&date_to=` → `[{...}]`（H5 ownership 校验） | 聊天记录搜索、数据面板 |
| `/api/v1/messages/{id}` | DELETE | —（撤回/删除，H5 ownership 校验） | 聊天页长按菜单 |
| `/api/v1/messages/{id}/recall` | PUT | —（H5 ownership 校验） | 聊天页撤回 |
| `/api/v1/messages/{id}/report` | POST | `{reason}` **（v2: reason 改可选——原型无 reason 输入 UI）** | 聊天页举报 |

- **S3 硬前置**：依赖 Phase 0.11 `chat_messages` 新表（`bot_id, channel, sender_id, content, role, client_id, message_type, status, retry_count`）——现有 `memories` 表无 client_id/bot_id 字段，无法做幂等去重
- **S1 流式实现**：新写 `call_deepseek_api_stream(messages) -> AsyncGenerator[str, None]`，`aiohttp` 流读 SSE `data: {...}` 帧逐 chunk yield；`api.py:79` 当前 `stream:False` 不可直接复用
- 复用：`api.py:call_deepseek_api`（非流式 fallback）+ `db_memories`（历史，需 Phase 0.11 新表）+ `media.py`（多媒体，**v2: media.py 依赖 MessageSegment 不可从 FastAPI 调用，需替换**）+ Pipeline（仅方案 A）
- 关键协议设计（对齐决策 F3/F5）：
  - **全局单 WS 连接**，帧带 `bot_id` 路由（切换 Bot 秒切）
  - **client_id 幂等去重**（UNIQUE INDEX `(bot_id, client_id)`，断网重发只存一条）
  - **server_id 权威时间源**（不信任手机时间）
  - 游标分页（非 offset），索引 `idx_messages_bot_created`
  - ACK + 心跳（30s ping/10s 超时）+ 重连
  - **H5 ownership**：WS 每条 msg 帧校验 bot_id 属于 JWT.user_id
- 前端触发：**2.4 聊天页 + 2.5 聊天记录 + 2.8 仪表盘**（整个 Phase 2 最关键触发点）
