## 一、项目定位

### 1.1 产品概述

一个 Android App，任何人都能下载注册，**创建属于自己的 AI Bot**。Bot 可通过以下三个通道与用户交互：

| 通道 | 说明 | 接入方式 |
|------|------|---------|
| 💬 App 内聊天 | App 内的直接对话界面 | WebSocket + LLM 流式 |
| 🐧 QQ | 绑定 QQ 号即可在 QQ 中使用 | NapCat + OneBot V11（MVP 共享号） |
| 💚 微信 | 绑定微信号即可在微信中使用 | ClawBot iLink（腾讯官方 API） |

### 1.2 用户能做什么

- 🤖 **创建 Bot**：选人格模板（傲娇/温柔/毒舌/元气/三无/腹黑），自定义名字/性格/口头禅
- 💬 **三通道聊天**：App 内直接聊、QQ 上聊、微信上聊，聊天记录统一互通
- ⚙️ **定制风格**：调整回复频率/详细度/主动度/幽默度
- 📊 **查看数据**：消息统计、Bot 情绪趋势、好友活跃排行
- 🔧 **高级用户**：自备 API Key 解锁无限消息/高级模型/语音/图片生成

### 1.3 开发者（你）能做什么

- 🖥️ 服务器资源监控（CPU/内存/带宽）
- 👥 全部用户和 Bot 管理
- 📋 实时系统日志
- 💰 Token 消耗/费用查看
- 🗄️ 一键备份/恢复

### 1.4 隐私与合规（App 上架前提）

| 事项 | 措施 |
|------|------|
| 手机号 | bcrypt 哈希 + 数据库层 AES-256 加密（双层保护） |
| API Key | 腾讯云 KMS 加密主密钥 → AES-256 加密存储 → 前端只显示后 4 位 |
| 聊天记录 | 用户可随时删除/导出；后台日志不记录消息正文 |
| 用户协议 & 隐私政策 | App 内展示，首次使用需明确同意 |
| 内容合规 | LLM 输出过滤敏感词；用户可举报不良内容 |
| ICP 备案 | APK 分发前确认备案要求 |

### 5.4 WebSocket 协议规范

> 前端决策 F3/F5/F6 依赖此规范。Phase 1 开工前冻结，前后端以此为共同契约。

#### 5.4.1 连接

```
ws[s]://host:8766/ws?token=<jwt_access_token>
```

- JWT 认证：URL query param 传递 access_token
- Token 过期：服务端发送 `{"type":"error","code":"token_expired"}` → 客户端 Dio Interceptor 自动刷新 → 新 Token 重连
- 心跳：服务端每 30s 发送 `{"type":"ping"}`，客户端 10s 内回复 `{"type":"pong"}`，超时断开
- 全局单连接：一个 WS 连接服务所有 Bot 切换，消息帧带 `bot_id` 路由

**JWT Refresh Token 轮换策略**：

```
1. 客户端用 access_token 连接 WS（URL query param）
2. 服务端返回 token_expired → 客户端 Dio Interceptor 自动用 refresh_token 换新
3. 每次刷新后：
 - 颁发新 access_token（15min TTL）+ 新 refresh_token（7天 TTL）
 - 标记旧的 refresh_token 为已使用（作废）
 - 旧 refresh_token 给 30s 宽限期（处理并发刷新竞争）
4. refresh_token 也过期 → 清除本地 Token → 跳转登录页
5. 并发刷新保护：多个 401 同时触发时，只发一次刷新请求，其余等待结果
```

> 轮换的意义：即使 refresh_token 被窃取，正常用户下次刷新时旧 token 立即作废，攻击者的窗口只有 7 天且与正常用户互斥。

#### 5.4.2 消息帧格式

**客户端 → 服务端（发送消息）**：
```json
{
 "type": "message",
 "bot_id": 1,
 "client_id": "uuid-v4-client-generated",
 "message_type": "text",
 "content": "今天天气怎么样？",
 "channel": "app"
}
```

**服务端 → 客户端（ACK 确认）**：
```json
{
 "type": "ack",
 "client_id": "uuid-v4-client-generated",
 "server_id": 12345,
 "status": "received"
}
```

**服务端 → 客户端（流式回复）**：
```json
{"type": "chunk", "bot_id": 1, "content": "今", "index": 0}
{"type": "chunk", "bot_id": 1, "content": "天", "index": 1}
...
{"type": "done", "bot_id": 1, "server_id": 12346, "total_chunks": 15}
```

**心跳**：
```json
{"type": "ping"}
{"type": "pong"}
```

#### 5.4.3 离线重连与幂等

```
1. 客户端生成 client_id（UUID v4）随每条消息发送
2. 服务端收到消息 → 检查 (bot_id, client_id) 唯一索引：
 - 不存在 → 存入 + 处理 + 返回 ACK
 - 已存在 → 直接返回原 ACK（幂等，不重复处理）
3. 客户端收到 ACK → 将 Hive 离线队列中该消息标记为 "acked" → 清除
4. 断网重连后：
 a. 客户端重连 WS
 b. 遍历 Hive 离线队列中所有 "pending" 消息 → 重新发送（同一 client_id）
 c. 服务端幂等去重，不会产生重复消息
 d. 拉取服务端最后一条 server_id 之后的新消息（增量同步）

**消息处理失败重试机制**：

```
时序问题：客户端发消息→服务端存 DB→回 ACK→客户端不重发→但 LLM 调用失败（超时/500）
→ 消息被存储但永远不被回复 ❌

解决方案：
1. chat_messages 表新增 status 字段（默认 pending）
 状态流转：pending → processing → replied → (failed)

2. 后台重试任务（APScheduler，每 30s 扫描）：
 扫描 status=failed AND created_at > NOW-5min AND retry_count<3
 → 重新送入 Pipeline → LLM → 回复
 → retry_count=3 仍未成功 → status=permanent_fail + 通知用户「消息发送失败，请重试」

3. 客户端侧：发送后 30s 未收到回复 → 显示「发送中...」→ 60s 仍无回复 → 显示「发送失败，点击重试」
```

> 防止静默丢消息：ACK 只代表「消息已安全存储」，不代表「已生成回复」。
```

#### 5.4.4 增量同步

```
GET /api/v1/messages?bot_id=1&after_server_id=12300&limit=50

响应：
{
 "messages": [...],
 "next_cursor": "12350",
 "has_more": true
}
```

- `after_server_id`：客户端本地最新的 server_id，拉取之后的新消息
- 首次加载（无本地缓存）：`GET /api/v1/messages?bot_id=1&limit=50`（不传 after_server_id）
- 游标分页使用 `(bot_id, created_at)` 复合索引保证性能

> 前后端共同契约。所有 API 错误响应统一格式：`{"code":"...","message":"..."}`。前端按 `code` 统一处理，不解析 `message`。

| F3 | WebSocket 策略 | **全局单连接**（App 启动即连，所有 Bot 复用） | 切换 Bot 秒切无延迟，消息协议带 `bot_id` 路由 |
| F4 | 流式渲染 | **AnimatedList 优先，CustomPainter 备选** | AnimatedList 实现简单，限 100ms 刷新一次足够流畅；如有性能问题再切 CustomPainter |
| F5 | 离线消息 | **(server_id, client_id) 双主键去重** + 服务器时间戳单一时间源 | 客户端预生成 UUID→服务端确认；增量拉取用游标，不管手机时间 |
| F6 | Token 刷新 | **Dio Interceptor 自动无感刷新** | 401→自动用 refresh_token 换新 access_token→重试原请求；并发只刷新一次；refresh 也过期→跳登录页 |
| F7 | minSdkVersion | **API 24（Android 7.0）** | 2026 年覆盖 95%+ 设备，WebSocket/Dio 原生兼容 |
| F8 | MVP 输入方式 | **文字 + 表情包 + 图片 + 语音 + 撤回** | 后端 STT/Vision 引擎已就绪（百度 STT 免费、GLM-4V 免费），只需包 API 层（Phase 1.14，1.5d） |
| F9 | 深色模式 | MVP 做（亮色/暗色自动切换） | flutter_gen_ai_chat_ui 自带暗色适配 |
| F10 | Markdown 渲染 | flutter_gen_ai_chat_ui 内置 + flutter_highlight 代码高亮 | 不额外引入包 |

#### 底部导航胶囊动画

导航栏使用**动态胶囊指示器**，包裹图标+文字，切换时拉伸拖影：

```
┌─────────────────────────────────────┐
│ [首页] [聊天] [Bot管理] [我的] │
│   └──████████──┘ ← 胶囊  │
│   ██████████ ← 拖影 t1 │
│   ████████  ← 拖影 t2 │
└─────────────────────────────────────┘
```

| 属性 | 值 |
|------|-----|
| 形状 | 圆角矩形 (border-radius: 14px)，包裹图标+文字 |
| 主体颜色 | 马卡龙软粉渐变: `rgba(244,114,182,0.8) → rgba(236,72,153,0.72) → rgba(219,90,130,0.65)` |
| 边框 | 1px `rgba(255,255,255,0.15)` + 内高光 `inset 0 1px 0 rgba(255,255,255,0.22)` |
| 投影 | `0 2px 14px rgba(244,114,182,0.28)` |
| 拖影 t1 | `rgba(244,114,182,0.3)` + blur 5px + lag 45ms |
| 拖影 t2 | `rgba(244,114,182,0.15)` + blur 10px + lag 90ms |
| 滑动曲线 | spring `cubic-bezier(0.34,1.56,0.64,1)` (left/top 380ms, width/height 420ms) |
| 拉伸倍率 | 1.5× 目标宽度，方向性偏差（右滑偏右 20% / 左滑偏左 80%） |
| 导航同步 | **回调链式**：Phase 2 压缩完成后 onComplete → 立即跳转，无独立定时器偏差 |

动画流程：`点击 → 方向性拉伸 1.5× + 滑向目标 (spring 380ms) → 压缩至正常宽度 (spring 420ms) → 拖影滞后跟随 (45/90ms) → onComplete 回调同步跳转 (总 800ms)`

实现文件：[`shared/components.css`](../安卓控制面板UI原型/shared/components.css) (`.nav-capsule` / `.nav-capsule-trail.t1/.t2`) + [`shared/app.js`](../安卓控制面板UI原型/shared/app.js) (`initBottomNav()` / `animateCapsule()`)

> 原型定位：**视觉方向稿**，快速对齐设计共识。不替代 Flutter 真实 UI 开发，仅为讨论和确认整体风格之用。已补齐登录/注册/隐私同意原型、用户协议/隐私政策/开源许可、管理员面板；色系统一到 v9 马卡龙 `#F472B6` 软粉色系，26 文件全部对齐。

### 10.5 表单验证规则

> 前后端共同契约。Phase 1 后端 + Phase 2 前端使用同一套规则，避免各写各的导致不一致。

| 字段 | 规则 | 前端行为 | 后端校验 |
|------|------|---------|:---:|
| 手机号 | 11 位数字，`1[3-9]\d{9}` | 输入时实时格式校验 + 失焦时格式提示 | ✅ 注册/登录时校验 |
| 密码 | 8-64 字符，含数字+字母（允许特殊字符） | 实时强度条：弱(红)/一般(黄)/强(绿)/很强(蓝) 4 段 | ✅ 注册/改密时校验 |
| 确认密码 | 与密码完全一致 | 实时比对，不一致标红 | ✅ 注册/改密时校验 |
| 昵称 | 1-20 字符，中英文/数字/下划线/Emoji | 字数计数，超限标红 | ✅ PATCH profile 时校验 |
| Bot 名字 | 1-12 字符，中英文/数字/常用符号，不含敏感词 | 字数计数 + 敏感词库前端预检 | ✅ 创建/改名时校验 |
| 口头禅 | 0-50 字符 | 字数计数 | ✅ 创建/改名时校验 |
| API Key | 根据厂商不同格式：DeepSeek `sk-` 开头 ~35字符 / OpenAI `sk-` 开头 ~51字符 / Kimi `sk-` 开头 | 前缀预检 + 长度范围校验 | ✅ 存入前校验格式 |
| 生日 | `YYYY-MM-DD` 或空，1900-01-01 ~ 今天 | 日期选择器（不早于1900，不晚于今天） | ✅ PATCH profile 时校验 |
| 性别 | `'male'` / `'female'` / `'other'` / `''` | 下拉选择 | ✅ PATCH profile 时校验 |

#### 10.5.1 密码强度算法

```
评分项：
 ≥8 字符 +1
 ≥12 字符 +1
 含数字  +1
 含小写字母 +1
 含大写字母 +1
 含特殊字符 +1

得分 → 强度：
 0-1 → 弱（红） — 禁止提交
 2-3 → 一般（黄）— 允许提交但提示
 4-5 → 强（绿） — 允许提交
 6 → 很强（蓝）— 允许提交
```

#### 10.5.2 敏感词过滤（Bot 名字）

维护一个轻量敏感词库（~500 词），前端预检 + 后端最终校验。命中时提示「Bot 名字含有不当内容，请修改」。

### 10.6 ShellRoute 导航架构决策

> 对应优化方案 O5。Phase 2.1 脚手架阶段即实施，避免后续返工。

**决策**：使用 GoRouter `ShellRoute` + `BottomNavigationBar` 实现底部 Tab 导航。

```
路由树：
/splash     → SplashScreen
/login      → LoginScreen
/register     → RegisterScreen
/onboarding    → OnboardingScreen

ShellRoute (BottomNavigationBar, 4 Tab):
 /home     → HomeScreen (首页仪表盘)
 /chat/:botId    → ChatScreen (聊天页)
 /bot/:botId/settings  → BotSettingsScreen (Bot 设置)
 /bots     → BotListScreen (我的 Bot)
 /data     → DataScreen (数据看板)

独立页面（不在 Tab 内）:
 /abilities/:botId  → AbilitiesScreen
 /notifications   → NotificationsScreen
 /settings    → SettingsScreen
 /settings/profile  → ProfileScreen
 /settings/change-password → ChangePasswordScreen
 /settings/data-permissions → DataPermissionsScreen
 /settings/blacklist  → BlacklistScreen
 /settings/terms   → TermsScreen
 /settings/privacy  → PrivacyScreen
 /settings/oss   → OssScreen
 /admin     → AdminScreen (仅 admin)
```

**Tab 切换与 WebSocket**：
- WebSocket 全局单连接（F3 决策），切 Tab 不断开
- `chatProvider` 在对应 `ChatScreen` 挂载时订阅 WS 消息
- `ChatScreen` 不可见时，`chatProvider` 继续接收消息更新未读计数
- 切 Tab 回到聊天页时，消息列表无需重新加载

**导航守卫**：
- GoRouter `redirect` 实现：
 - 未登录 → 所有需登录路由 → `/login`
 - 已登录 + 无 Bot → `/onboarding`
 - 已登录 + 有 Bot + 访问 `/login` → `/home`
 - 非 admin + 访问 `/admin` → `/home`
